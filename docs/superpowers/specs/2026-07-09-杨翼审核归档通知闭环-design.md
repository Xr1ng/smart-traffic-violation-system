## 1. 背景与目标

Plan 1 完成了杨翼-1~4（骨架/DB/图片接入/文件存储）+ 张浩-1~2（JWT/RBAC）+ 唐高鹏-1（建表），即后端地基。本 spec 实现杨翼剩下的核心业务闭环：审核工作台用的案件管理 API、人工审核工作流（通过/驳回/重审）、违章归档、邮件通知、随手拍奖励。

**关键依赖约束**：本闭环依赖 Plan 2 的 AI 管线（YOLO/OCR/规则/LLM）把案件从 `uploaded` 推到 `pending_human_review`，以及 Celery 做异步通知。Plan 2 尚未做，故本 chunk 采用**接口先行 + 隔离测试**：AI 相关字段返回可空、`request-recheck` 做成 stub、通知同步发送、Celery 任务留 Plan 2。端到端联调在 Plan 2 完成后接上。

## 4. 新模型（Plan 1 没有，本 chunk 新增 + Alembic 迁移）

```text
Vehicle
  id, plate_no(unique, index), owner_id→users, vehicle_type, color, created_at

Violation
  id, violation_no(unique, index), case_id→cases, vehicle_id→vehicles(nullable),
  owner_id→users(nullable), plate_no, violation_type, occurred_at, location_text,
  fine_amount, points, status(pending/handled/cancelled), created_at

Notification
  id, violation_id→violations, owner_id, channel(email), recipient, content,
  status(pending/sent/failed), provider, provider_msg_id, retry_count, sent_at, created_at

NotificationTemplate
  id, code(unique), channel(email), subject_template, body_template, status, created_at

Reward
  id, citizen_id→users, case_id→cases, violation_id→violations(nullable),
  amount, reason, status(pending/paid), created_at

AuditLog
  id, actor_id→users(nullable), action, target_type, target_id, detail, ip, created_at
```

> `AuditLog` 用于 approve/reject 等审核动作留痕（§10）。

> Plan 1 已有的 `Case` 表已含 `reviewer_id/review_opinion/reviewed_at/violation_type/plate_no`，approve 直接写。`violations`/`notifications`/`notification_templates` 对齐系统设计方案 §6；`rewards` 为本 chunk 新表（spec 原无，为奖励子模块）。

## 5. 案件状态流

```text
uploaded ──[Plan 2 AI]──> detecting ──> ai_reviewing ──> pending_human_review
                                                              │ approve        │ reject
                                                              ▼                ▼
                                                  建 Violation+Reward       rejected(记原因)
                                                      +发 Notification
                                                              ▼
                                                          notified
```

- `approved`/`archived` 在 approve 内是瞬态（同步发通知，approve 直接 `pending_human_review → notified`，`archived` 在事务内设过）。
- 本 chunk 无 Plan 2 AI，测试用例手动置 `pending_human_review` 测 approve/reject。
- `request-recheck`：stub，返回 202「AI 复核已排队（Plan 2 接入后生效）」，不触发真实 AI 链。

## 6. 接口设计

统一前缀 `/api/v1`。

### 6.1 案件管理（杨翼-5）

**`GET /cases`** — 列表，分页+筛选+角色作用域

```text
查询参数: status, source_type, location_text, plate_no, start_time, end_time, page(默认1), page_size(默认20)
角色作用域:
  citizen  → 只看 source_id = 自己（= 举报进度查询，杨翼-9）
  reviewer → 看全部（默认 pending_human_review 优先排序）
  admin    → 看全部
  camera   → 403
返回: {items:[{case_no, status, source_type, plate_no, violation_type, captured_at, location_text}], total, page, page_size}
```

**`GET /cases/{id}`** — 详情（AI 字段可空）

```text
返回: case + intake_event + media_assets(original/annotated/cropped_plate)
      + ai_detection_result: null | {objects,...}        ← Plan 2 填
      + violation_rule_result: null | {...}              ← Plan 2 填
      + ai_review_result: null | {conclusion, reason}    ← Plan 2 填
      + review: {reviewer_id, review_opinion, reviewed_at}
角色: citizen 只看自己；reviewer/admin 看全部
```

### 6.2 审核工作流（杨翼-6）

**`POST /cases/{id}/approve`** — 闭环枢纽

```text
请求: {plate_no, violation_type, fine_amount, points, review_opinion}
逻辑:
  1. 校验 case.status == pending_human_review（否则 409）
  2. 写 case: plate_no/violation_type/reviewer_id/review_opinion/reviewed_at
  3. 查 Vehicle by plate_no → owner_id → User.email（查不到 owner_id=null）
  4. 建 Violation(violation_no=VIO{YYYYMMDD}{id:06d}, case_id, vehicle_id, owner_id, plate_no,
     violation_type, fine_amount, points, occurred_at=captured_at, location_text, status=pending)
  5. 若 source_type=citizen → RewardService.grant_reward(citizen_id=source_id, case_id, violation_id, violation_type)
  6. NotificationService.send_violation_notification(violation) → 建 Notification(sent/failed)
  7. case.status = notified
返回: {violation_no, notification_status, reward_id?}
```

**`POST /cases/{id}/reject`**

```text
请求: {reject_reason}
逻辑: 校验 pending_human_review；case.status=rejected；review_opinion=reject_reason；reviewer_id/reviewed_at
返回: {case_no, status: rejected}
```

**`POST /cases/{id}/request-recheck`** — stub

```text
逻辑: 校验 pending_human_review；返回 202 {message:"AI 复核已排队（Plan 2 接入后生效）"}
不触发真实 AI 链。
```

### 6.3 违章查询（杨翼-7 附带）

```text
GET /violations           — 列表，按 plate_no/violation_type/location_text/start_time/end_time/status 筛选
GET /violations/{id}      — 详情
GET /owners/{owner_id}/violations — 车主本人违章（citizen 可查自己）
```

## 7. 通知模块（杨翼-8）

```text
NotificationProvider (抽象, app/services/notification_provider.py):
  send(to_email: str, subject: str, body: str) -> SendResult(status, provider_msg_id)
├─ EmailSmtpProvider   # 真实 SMTP，stdlib smtplib，同步
└─ FakeNotificationProvider  # 测试，记录不真发

NotificationTemplate: code=violation_notice, channel=email, subject_template, body_template
  渲染变量: plate_no, violation_type, fine_amount, points, location_text, occurred_at, violation_no

NotificationService.send_violation_notification(violation, owner_email):
  载入 violation_notice 模板 → 渲染 → provider.send → 建 Notification(violation_id, owner_id, channel=email,
  recipient=owner_email, content, status=sent/failed, provider, provider_msg_id, sent_at)
  无 owner_email → Notification status=failed(no_recipient)
```

依赖注入：通过 `Depends` 注入 `NotificationProvider`，测试用 Fake，运行用 EmailSmtp。

## 8. 奖励模块（杨翼-9）

```text
RewardService.grant_reward(citizen_id, case_id, violation_id, violation_type) -> Reward:
  amount = REWARD_AMOUNTS.get(violation_type, REWARD_DEFAULT_AMOUNT)  # 先硬编码 dict
  建 Reward(citizen_id, case_id, violation_id, amount, reason=f"举报{violation_type}成立", status=pending)
  返回 Reward
举报进度查询 = 市民 GET /cases（角色作用域已覆盖，无单独端点）
```

## 9. 车辆-车主链路（杨翼-7）

```text
Vehicle 模型 + seed（几辆车 + 车主 + email，供闭环演示）
approve: 查 Vehicle by plate_no → owner_id → User.email
  查不到车: violation 建(owner_id=null)，Notification=failed(no_recipient)，case→notified
  查到车无邮箱: 同上 failed
  查到车有邮箱: 发邮件 → Notification(sent)
violation 是归档的 source of truth，通知 best-effort，不阻塞归档。
```

## 10. 错误处理

| 情况 | 处理 |
| --- | --- |
| approve/reject 非 `pending_human_review` | 409 |
| approve 缺 plate_no | 400 |
| 查不到车辆/车主邮箱 | violation 照建，Notification=failed，case→notified |
| 邮件发送失败 | Notification=failed，case→notified（手动/Celery 重试留 Plan 2） |
| citizen 访问他人案件 | 403 |
| .env 缺 SMTP 配置 | EmailSmtpProvider 发送时 Notification=failed，不崩 |
| 审计 | approve/reject 写 audit_logs（杨翼的审核动作留痕；audit_logs 表本 chunk 加） |

## 14. 验收标准

- `GET /cases` 支持分页+筛选+角色作用域（市民只看自己、camera 403、reviewer/admin 全量）。
- `GET /cases/{id}` 返回 case+intake+media+AI 字段（当前 null）+review 字段，权限隔离生效。
- `POST /cases/{id}/approve` 在 pending_human_review 案件上：建 Violation（含 violation_no）、source=citizen 时建 Reward、发邮件并建 Notification、case→notified。
- approve 缺 plate_no → 400；非 pending → 409；查不到车 → violation 照建 + Notification=failed。
- `POST /cases/{id}/reject` → case=rejected + 记原因；非 pending → 409。
- `POST /cases/{id}/request-recheck` → 202 stub。
- `GET /violations`、`/violations/{id}`、`/owners/{id}/violations` 可查。
- 邮件用 EmailSmtpProvider 真实可发（env 配置后），测试用 Fake。
- 审核动作写 audit_logs。
- 核心逻辑 TDD，测试全绿，AI 字段 null 断言。

## 15. 后续衔接（Plan 2 联调点）

- Plan 2 AI 链跑通后，案件自动 `uploaded→...→pending_human_review`，approve 端到端闭环成立。
- `request-recheck` 在 Plan 2 接真实 AI 重审链。
- `case 详情` 的 AI 字段由 Plan 2 填充（detection/rule/ai_review）。
- Plan 2 搭 Celery 后，`NotificationService.send` 包成 `send_notification_task` 异步发。
